# ADR-006: Sistema de notificaciones por email

**Estado:** ACCEPTED
**Fecha:** 2026-06-24
**Autores:** Equipo ADSO 3145555
**Equipos involucrados:** Arquitectura, Producto

---

## Contexto

El sistema Horarios SENA necesita notificar a los instructores y coordinadores sobre eventos importantes:

- Asignación de horarios.
- Cancelación de horarios.
- Observaciones críticas.
- Vencimiento de fechas de proyectos.

## Decisión

Se implementa un sistema de notificaciones por email usando un worker asíncrono.

### Detalles

- Framework: Spring Mail + JavaMailSender.
- Worker: Un microservicio notification-service que procesa eventos y envía emails.
- Cola: Se usa un broker de mensajes (RabbitMQ o Kafka) para desacoplar la generación del evento del envío del email.
- Plantillas: Se usan plantillas Thymeleaf para los emails (HTML con formato).
- Eventos que disparan notificaciones: SchedulePlanned, ScheduleCancelled, ObservationCreated, ProjectDeadlineApproaching.

## Consecuencias

### Positivas

- Desacoplamiento: el envío de emails no bloquea la operación principal.
- Resiliencia: si el servicio de email falla, el evento se puede reintentar.
- Escalabilidad: el worker puede escalar horizontalmente.

### Negativas / Trade-offs

- Complejidad adicional (broker de mensajes, worker).
- Latencia: el email no se envía instantáneamente (asíncrono).

### Riesgos

- El broker de mensajes debe ser configurado correctamente (persistencia, dead-letter queues).
- Las credenciales de SMTP deben ser gestionadas de forma segura.

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| Envío síncrono desde el servicio | Bloquea la operación principal y aumenta el tiempo de respuesta. |
| No usar notificaciones | No cumple con los requisitos de usuario. |
| Usar servicios externos (SendGrid, AWS SES) | No se justifica para el MVP; se puede integrar después. |
