# ADR-001: Clean Architecture para el backend

**Estado:** ACCEPTED
**Fecha:** 2026-06-24
**Autores:** Equipo ADSO 3145555
**Equipos involucrados:** Arquitectura, Desarrollo

---

## Contexto

El sistema Horarios SENA necesita una arquitectura que permita:

- Mantener la lógica de negocio independiente de frameworks y bases de datos.
- Facilitar las pruebas unitarias de la lógica de negocio.
- Permitir cambiar de tecnología (base de datos, framework web) sin afectar el dominio.
- Escalar el sistema a medida que crecen los requisitos.

El equipo evaluó diferentes enfoques arquitectónicos para el backend.

## Decisión

Se adopta Clean Architecture (Arquitectura Limpia) para el backend en Java.

La estructura de capas es la siguiente, de afuera hacia adentro:

- Capa Transport (HTTP): Controllers, DTOs, GlobalExceptionHandler.
- Capa Application (Use Cases): Services con lógica de negocio, Ports (interfaces).
- Capa Domain (Core): Entities, Value Objects, Domain Services, Domain Events.
- Capa Infrastructure (Persistence): Implementaciones de repositorios, mapeo JPA, configuración de BD.

### Reglas de dependencia

- Domain no depende de nada (solo Java estándar).
- Application depende de Domain y de interfaces definidas en Infrastructure.
- Transport depende de Application y Domain.
- Infrastructure depende de Domain y Application (implementa interfaces).

## Consecuencias

### Positivas

- Alta testeabilidad: la lógica de negocio se prueba sin infraestructura.
- Independencia de frameworks: se puede cambiar de base de datos o framework web sin afectar el dominio.
- Código más mantenible y comprensible.
- Facilita la evolución del sistema a largo plazo.

### Negativas / Trade-offs

- Curva de aprendizaje inicial para el equipo.
- Mayor cantidad de archivos y carpetas (más estructura).
- Puede sentirse como "overengineering" para funcionalidades simples.

### Riesgos

- El equipo debe entender bien los principios de Clean Architecture.
- La inversión de dependencias (DIP) debe aplicarse correctamente.

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| Arquitectura en capas tradicional | No separa claramente el dominio de la infraestructura, dificultando las pruebas. |
| Arquitectura hexagonal | Muy similar a Clean Architecture, pero Clean es más explícita en sus capas. |
| DDD (Domain-Driven Design) | Excesivo para el alcance actual; puede aplicarse gradualmente dentro de Clean Architecture. |
| CQRS | Complejo para el MVP; no hay necesidad asimétrica de lectura/escritura. |
