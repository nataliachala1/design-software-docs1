# ADR-007: Generación de reportes asíncronos

**Estado:** ACCEPTED
**Fecha:** 2026-06-24
**Autores:** Equipo ADSO 3145555
**Equipos involucrados:** Arquitectura, Producto

---

## Contexto

El sistema Horarios SENA necesita generar reportes en PDF y Excel de:

- Horarios por instructor, ambiente y ficha.
- Ocupación de ambientes.
- Carga horaria de instructores.
- Avance de proyectos formativos.

Los reportes pueden ser pesados (muchos datos) y consumir tiempo de procesamiento.

## Decisión

La generación de reportes se realiza de forma asíncrona con un worker dedicado.

### Detalles

- Worker: pdf-worker y report-worker.
- Tecnología: iText o Apache POI para generación de PDF/Excel.
- Cola: Se usa un broker de mensajes (RabbitMQ) para encolar las solicitudes de reportes.
- Almacenamiento: Los reportes generados se almacenan en el sistema de archivos o S3 y se devuelve una URL.
- Notificación: Cuando el reporte está listo, se notifica al usuario (por email o dashboard).

## Consecuencias

### Positivas

- No bloquea la API: el usuario puede continuar usando el sistema.
- Escalable: se pueden tener múltiples workers.
- Resiliencia: si falla la generación, se puede reintentar.

### Negativas / Trade-offs

- Complejidad adicional (broker de mensajes, worker, almacenamiento de archivos).
- El usuario no obtiene el reporte inmediatamente (debe esperar).

### Riesgos

- El almacenamiento de archivos debe ser gestionado (limpieza de reportes antiguos).
- Los reportes grandes pueden consumir mucha memoria.

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| Generación síncrona | Bloquea la API para reportes grandes. |
| No usar reportes | No cumple con los requisitos de usuario. |
| Usar servicios externos (Google Docs API) | No se justifica para el MVP. |
