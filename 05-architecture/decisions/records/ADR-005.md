# ADR-005: Microservicios con base de datos por servicio

**Estado:** ACCEPTED
**Fecha:** 2026-06-24
**Autores:** Equipo ADSO 3145555
**Equipos involucrados:** Arquitectura, DevOps

---

## Contexto

El sistema Horarios SENA está organizado en dominios que se implementarán como microservicios. Cada microservicio tiene responsabilidades específicas y debe ser desplegable independientemente.

Se requiere decidir cómo gestionar la persistencia de datos en un entorno de microservicios.

## Decisión

Cada microservicio tiene su propia base de datos (esquema) en PostgreSQL, sin compartir tablas con otros servicios.

### Detalles

- Esquemas: iam_schema, reference_schema, academic_schema, scheduling_schema, actors_schema, audit_schema.
- Comunicación: Los servicios se comunican por REST (síncrono) o eventos (asíncrono) para obtener datos de otros servicios.
- Propiedad: Cada servicio es el único que puede escribir en su propia base de datos.
- Consultas: Si un servicio necesita datos de otro, los consulta vía API, no directamente a la base de datos.

## Consecuencias

### Positivas

- Desacoplamiento: cada servicio es independiente en su persistencia.
- Escalabilidad: cada base de datos puede escalarse según la carga del servicio.
- Seguridad: límites claros entre dominios (bounded contexts).
- Flexibilidad: cada servicio puede elegir su propio esquema de base de datos.

### Negativas / Trade-offs

- Mayor complejidad en consultas que requieren datos de múltiples servicios (se deben hacer llamadas REST encadenadas).
- Consistencia eventual en lugar de transacciones distribuidas.
- Mayor latencia en operaciones que involucran varios servicios.

### Riesgos

- Las transacciones distribuidas son complejas (se debe usar el patrón Saga).
- Posible inconsistencia de datos si un servicio falla después de actualizar otro.

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| Base de datos única compartida | Rompe el desacoplamiento de microservicios. |
| Base de datos por servicio con replicación | Complejo y no necesario para el alcance actual. |
| Event sourcing | Demasiado complejo para el MVP. |
