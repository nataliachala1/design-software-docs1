# ADR-002: Autenticación con JWT y Spring Security

**Estado:** ACCEPTED
**Fecha:** 2026-06-24
**Autores:** Equipo ADSO 3145555
**Equipos involucrados:** Arquitectura, Seguridad

---

## Contexto

El sistema Horarios SENA necesita autenticación y autorización para controlar el acceso a los recursos. Se requiere:

- Autenticación de usuarios (Coordinadores, Instructores, Administradores).
- Control de acceso basado en roles (RBAC).
- Sesiones stateless para facilitar la escalabilidad.

## Decisión

Se adopta JWT (JSON Web Token) con Spring Security para autenticación y autorización.

### Detalles

- Autenticación: El usuario envía credenciales (usuario/contraseña) y recibe un JWT.
- Autorización: Cada endpoint verifica el rol del usuario usando @PreAuthorize.
- Almacenamiento: El JWT se almacena en el cliente (localStorage o cookie HttpOnly).
- Expiración: El token expira después de 8 horas.
- Roles: ROLE_COORDINADOR, ROLE_INSTRUCTOR, ROLE_ADMIN.
- Permisos: Se controlan con anotaciones de Spring Security.

### Flujo de autenticación

1. El cliente envía usuario y contraseña al endpoint de login.
2. El backend valida las credenciales y genera un JWT.
3. El backend devuelve el JWT al cliente.
4. El cliente incluye el JWT en el header Authorization: Bearer en cada petición.
5. El backend valida el JWT y verifica los roles antes de procesar la petición.
6. El backend devuelve la respuesta.

## Consecuencias

### Positivas

- Stateless: no se necesita almacenar sesiones en el servidor.
- Escalable: cualquier instancia puede validar el token.
- Estandarizado: JWT es ampliamente soportado.

### Negativas / Trade-offs

- Los tokens no pueden ser revocados fácilmente (se usa expiración corta).
- El tamaño del token puede ser grande si se incluyen muchos claims.
- Requiere almacenamiento seguro en el cliente (HttpOnly cookie recomendado).

### Riesgos

- Si el secreto JWT se compromete, todos los tokens son vulnerables (rotar secreto).
- Los tokens expuestos pueden ser usados para ataques (uso de HTTPS obligatorio).

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| Sesiones basadas en cookies | Requiere almacenamiento en servidor y no es stateless. |
| OAuth2 / OpenID Connect | Complejo para el MVP; se puede integrar después. |
| API Keys | No permite autorización granular basada en roles. |
